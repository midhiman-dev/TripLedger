How to read this backlog

This backlog is intentionally written in human language so product, design, engineering, and QA can all use the same artifact.

Must Have Should Have Could Have User Story Enabler

  • User Story items describe visible product behavior.
  • Enabler items capture technical work required to make the user promise real.
  • Acceptance Snapshot gives the essence of “done” without becoming a full test script.

MVP outcome this backlog is driving toward

A road-trip group should be able to create a trip, set budgets, log expenses without signal, and regain a fully reconciled shared ledger when connectivity returns.

  • Offline is the default operating mode.
  • Expense logging must feel faster than typical online apps.
  • Data loss is unacceptable; conflicts must be surfaced, not hidden.
  • PWA installability must remove app-store friction.

6. Epic 5 — Sync, Offline Queue & Conflict Resolution

Goal

Deliver the hardest promise in the product: multiple devices can work independently offline and still converge on a trustworthy shared ledger when connectivity returns.
Epic Priority: Must Have Highest Technical Risk Outcome: Shared ledger reconciles safely
IDTypeBacklog ItemPrioritySizeAcceptance Snapshot
PB-401EnablerCreate a durable local pending queue that records unsynced creates, edits, deletes, and join actions without losing order or identity.MustLPending items survive app refreshes, restarts, and temporary failures.
PB-402EnablerImplement a pull-then-push sync loop so the client first ingests remote changes, then submits local pending changes.MustLDevices converge cleanly after reconnect instead of blindly overwriting each other.
PB-403EnablerSupport Android/Chrome Background Sync and an iOS-safe foreground fallback on app focus or reconnect.MustMQueued records start syncing within the agreed reconnect window on both major mobile platforms.
PB-404EnablerImplement field-level HLC merge rules so independent edits on different fields can merge automatically.MustLNon-overlapping edits resolve without unnecessary user intervention.
PB-405User StoryAs a trip organiser, I want conflicting edits to be surfaced clearly with both versions shown side by side, so that I can make an informed choice instead of losing data silently.MustMConflict card shows both versions, their origins, and clear winner actions.
PB-406EnablerArchive the rejected conflict version and restore the winning record to synced state after resolution.MustMNo conflict disappears without trace; the ledger becomes stable again after the user decides.
PB-407User StoryAs any traveller, I want my queued expenses to sync automatically when signal returns, so that I do not have to remember to manually push data after every dead zone.MustSPending items sync without requiring a refresh button or manual command.
Why this epic is special: this is the core moat of TripLedger. It should be treated as early-cycle work, not late integration work. If this epic is weak, the rest of the product promise collapses.

7. Epic 6 — Security, Data & Release Readiness

Goal

Make the MVP safe to release by grounding it in a secure data model, region-appropriate hosting, operational telemetry, and realistic test coverage for offline behavior.
Epic Priority: Must Have Release Safeguards Outcome: MVP is shippable and supportable
IDTypeBacklog ItemPrioritySizeAcceptance Snapshot
PB-501EnablerModel the core cloud and local entities for trips, categories, expenses, memberships, tombstones, and conflict metadata.MustMData model supports offline creation, joining, sync, and conflict preservation without redesign.
PB-502EnablerEnable Supabase Row Level Security and membership-aware policies before beta launch.MustMOne trip’s data cannot be read or modified by a non-member device/session.
PB-503EnablerMake sync writes idempotent so retries or duplicate deliveries do not create duplicate expenses or ledger corruption.MustMRepeated submission of the same mutation returns a stable outcome.
PB-504EnablerAdd operational logging and alerts for sync failures, data-loss indicators, and unusually high conflict rates.MustMReliability issues are visible to the team during beta, not discovered only by users.
PB-505EnablerCreate storage-quota and local-retention safeguards so low-end devices do not fail silently when IndexedDB pressure rises.ShouldMThe app can warn users and preserve unsynced records first if storage becomes constrained.
PB-506EnablerBuild a QA checklist and automated regression coverage for offline launch, trip setup, quick-add, reconnect sync, and conflict resolution.MustMCore paths are repeatably testable before beta and before public launch.

9. Suggested Delivery Sequence

The backlog is most likely to succeed if built in the order below. This sequencing follows the product’s risk profile, not just UI visibility.

Step 1
Foundation
PWA shell, service worker, local database, baseline state model.
Step 2
Trip Setup
Create trip, categories, share code, local persistence.
Step 3
Quick Add
3-tap expense flow with optimistic UI and pending state.
Step 4
Dashboard
Totals, category variance, history, settlement base.
Step 5
Sync Engine
Pull/push sync, queue replay, Android and iOS reconnect handling.
Step 6
Conflicts & QA
Conflict UI, RLS hardening, telemetry, beta release testing.

What “MVP done” should mean

  • A trip can be created and shared from one device.
  • Multiple devices can log expenses without signal.
  • The dashboard remains trustworthy offline.
  • Pending records sync automatically after reconnect.
  • Conflicts are surfaced and resolvable without data loss.
  • The app is installable, testable, supportable, and safe for beta.
