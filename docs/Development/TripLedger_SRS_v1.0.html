1. Introduction

1.1 Purpose

TripLedger is an offline-first Progressive Web App for Indian road-trip groups to create shared trip budgets, log expenses without connectivity, and reconcile data when network access returns. The software shall prioritize fast local interactions, durable local storage, and conflict-aware synchronization over always-online assumptions.

This SRS converts product intent into technical requirements that engineering, QA, and architecture teams can implement and verify.

1.2 Scope

The MVP covers trip creation, category budgeting, 3-tap expense logging, local-first dashboarding, deferred trip joining, PWA installation, background/foreground sync, and conflict resolution.

Receipt upload, GPS tagging, multi-currency, advanced analytics, exports, UPI settlement, and native mobile apps are outside MVP scope.

1.3 Product Objectives and Engineering Translation

Business ObjectiveTechnical Implication
Offline reliability as product moatAll core reads/writes shall execute locally first. No critical flow shall block on a remote call.
Fast, low-friction usage in travel conditionsExpense logging shall complete in ≤ 3 taps with optimistic UI and no loading spinner.
Zero data loss benchmarkSoft deletes, immutable client IDs, durable pending queues, and explicit conflict workflows are mandatory.
Install without app-store frictionThe app shall meet installable PWA requirements, launch in standalone mode, and support offline shell caching.

1.4 Definitions and Technical Terms

TermDefinition
Offline-firstDesign principle where local device operation is the default mode and network connectivity is treated as an enhancement rather than a prerequisite.
PWAProgressive Web App installable from the browser, capable of standalone launch, local caching, and service-worker-driven offline behavior.
IndexedDBBrowser local database used to persist trips, categories, expenses, membership state, and sync metadata on device.
Dexie.jsIndexedDB wrapper used to model tables, transactions, and reactive data access on the client.
HLCHybrid Logical Clock represented as { wallClock, logical, nodeId } and used for deterministic ordering during sync and merge.
ConflictA true merge collision where the same field of the same record is edited offline on two devices and cannot be safely auto-merged.
RLSRow Level Security policies in Supabase/Postgres that restrict data visibility and mutation by membership and device/session context.

2. System Architecture

2.1 Architectural Style

The system shall be implemented as a client-heavy, offline-first SPA using Vite + React for presentation, Zustand for local UI state, Dexie.js/IndexedDB for durable local data, Workbox service workers for shell caching and sync orchestration, and Supabase for cloud persistence, synchronization endpoints, and row-level protected data access.

Figure 1. Logical component architecture for the TripLedger MVP.

2.2 Runtime Behavior

ModeExpected Behavior
Cold start, onlineLoad cached shell if available, hydrate from IndexedDB, then refresh from cloud where needed.
Cold start, offlineLoad cached shell within target time, use IndexedDB as sole source of truth, surface offline banner and pending state.
User write while offlineWrite to IndexedDB transactionally, generate UUID and HLC locally, update UI immediately, add queue entry for sync.
Reconnect on Android/ChromeService worker sync event shall replay queued mutations and fetch deltas.
Reconnect on iOS/SafariForeground poller on app focus or connectivity change shall trigger queued sync because Background Sync API support is limited.
Conflict detectedDo not silently overwrite. Mark record as CONFLICT, archive both variants, and route to conflict resolution UI.
 
SYNC SEQUENCE (AUTHORITATIVE FLOW)

1. User action on device
   → validate locally
   → write record to IndexedDB
   → set sync_status = PENDING
   → attach HLC + deviceId + UUID
   → update dashboard immediately

2. Connectivity restored
   → client/SW requests server changes since last_sync_hlc
   → client merges server deltas field-by-field
   → client pushes pending local mutations

3. Server merge
   → append non-conflicting new records
   → merge same record / different field by HLC
   → same field / same ordering ambiguity => create conflict payload

4. Client finalization
   → PENDING becomes SYNCED for committed records
   → conflicting records become CONFLICT
   → sync indicator updates globally

4.2 Capacity and Scalability

The source materials define growth targets in terms of active installs rather than strict simultaneous concurrency ceilings. This SRS therefore specifies scaling objectives aligned to the MVP business plan and managed-service architecture.

IDRequirement
NFR-S-01The MVP architecture shall support an initial closed beta of 100 users and a path to 5,000 active installs within 90 days without re-platforming.
NFR-S-02The backend design shall support typical trip groups of 2–6 participants and shall not hard-code assumptions that prevent future higher participant counts.
NFR-S-03Application hosting and database layers shall be horizontally scalable through Vercel and Supabase service tier upgrades rather than bespoke infrastructure changes.
NFR-S-04Sync processing shall be idempotent so retries, duplicate delivery, or app restarts do not create duplicate records or ledger corruption.
NFR-S-05Local query performance shall remain acceptable with at least 500 expenses per trip and 90 days of retained synced history on device.

4.3 Security

IDRequirement
NFR-SEC-01All network traffic between client and backend shall use HTTPS/TLS 1.2+ with HSTS enabled at the hosting layer where supported.
NFR-SEC-02All shared data tables in Supabase shall enforce Row Level Security before public beta launch.
NFR-SEC-03MVP access control shall be account-light: device identity and trip membership shall authorize access, and no email/password onboarding shall be required.
NFR-SEC-04The backend shall validate that a device/session can only read or mutate trips for which it has a valid membership association.
NFR-SEC-05The system shall minimize data collection and avoid unnecessary PII. Analytics shall use anonymous identifiers and activate only after consent.
NFR-SEC-06Local persistent data shall be scoped to browser origin. The implementation shall warn users that device/browser clearing can remove unsynced local data.

6.2 Data Flow

CREATE / UPDATE FLOW
UI action
  → input validation
  → domain service generates UUID + HLC
  → Dexie transaction writes record + queue metadata
  → Zustand recomputes totals and sync state
  → Service worker or foreground poller submits queued mutation
  → Supabase stores authoritative copy and returns result
  → local status becomes SYNCED or CONFLICT

READ FLOW
screen opens
  → read from IndexedDB immediately
  → render local state
  → if online, fetch remote deltas
  → merge by HLC
  → refresh visible state

7. Error Handling & Recovery

7.1 Error Classes

Error ClassExamplesRequired Response
Validation errormissing trip name, amount = 0, malformed trip codeBlock submit, show inline guidance, do not create local dirty record.
Offline network absenceno signal, fetch timeout, background sync unavailableContinue local operation, show Offline/Pending UI, queue sync for retry.
Merge conflictsame field edited on multiple devicesCreate explicit conflict state and preserve both versions.
Authorization errordevice not part of trip, RLS rejectionStop remote action, preserve local queue, prompt rejoin/re-auth pathway.
Storage quota errorIndexedDB quota exceededWarn user, stop non-essential caching, preserve most recent pending data first.
Unexpected crashrender exception, service worker failure, parsing bugCapture in Sentry, preserve local database, allow safe app restart.

7.2 Recovery Requirements

IDRequirement
ERR-01Any network failure during a mutation shall leave the locally created record intact and in PENDING state.
ERR-02Sync operations shall be retry-safe; replay of the same mutation shall not duplicate an expense or category.
ERR-03If the service worker is unavailable or unsupported for background replay, the app shall retry in the foreground when opened or focused.
ERR-04If merge logic cannot determine an automatic winner, the system shall prefer a surfaced conflict over silent overwrite.
ERR-05An Undo affordance should be presented after delete while the record remains recoverable as a soft-delete tombstone.
ERR-06Crash recovery shall prioritize rehydrating the last durable IndexedDB state on restart before any remote synchronization occurs.
ERR-07The app shall maintain an operational sync log locally for debugging support scenarios without exposing internal details to end users by default.

8. Assumptions & Out of Scope

8.1 Key Assumptions

  • Trip participation is intentionally lightweight; MVP avoids full account onboarding.
  • Cloud authorization is enforced through membership-aware backend policies even when the UI feels accountless.
  • HLC-based field merges and manual conflicts are preferred over CRDT complexity for this domain.
  • India is the primary market for MVP, and INR is the default operating currency.
  • Typical groups contain 2–6 travelers, but the data model should not hard-code this as an absolute limit.

8.2 Explicitly Out of Scope for MVP

  • Receipt photo capture and blob synchronization
  • GPS-based location tagging
  • Multi-currency and live exchange rates
  • Advanced charts and post-trip analytics
  • PDF/CSV export
  • UPI-integrated settlement
  • Native iOS/Android apps
  • International travel support
